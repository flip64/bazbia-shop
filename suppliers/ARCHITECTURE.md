# معماری ماژول Suppliers

## هدف

ماژول `suppliers` مسئول ارتباط با تأمین‌کنندگان، دریافت اطلاعات محصولات، استانداردسازی داده‌ها و همگام‌سازی آن‌ها با دیتابیس فروشگاه بازبیا است.

این سند تصمیم‌های قطعی معماری این ماژول را ثبت می‌کند تا توسعه‌های بعدی با منطق یکسان انجام شوند.

---

# ساختار کلی جریان داده

```text
Website / API / File
        ↓
Fetcher
        ↓
Extractor
        ↓
Standard Data
        ↓
Sync
        ↓
Services
        ↓
Django Models
```

هر لایه فقط یک مسئولیت مشخص دارد و نباید وارد مسئولیت لایه‌های دیگر شود.

---

# 1. Fetchers

## مسئولیت

برقراری ارتباط با سایت یا منبع داده تأمین‌کننده و دریافت اطلاعات خام محصولات.

این بخش تنها جایی است که با اینترنت، HTML، API، فایل و ساختار اختصاصی تأمین‌کننده سروکار دارد.

## ورودی‌های احتمالی

* آدرس محصول
* آدرس دسته‌بندی
* API تأمین‌کننده
* فایل CSV، Excel یا JSON
* Sitemap
* فایل‌های ذخیره‌شده قبلی

## خروجی

داده خام یا اطلاعاتی که برای استخراج داده استاندارد استفاده می‌شود.

## مجاز است

* ارسال Request
* دریافت Response
* استفاده از `requests`
* استفاده از `BeautifulSoup`
* خواندن HTML یا API
* خواندن فایل‌های اختصاصی تأمین‌کننده
* مدیریت خطاهای ارتباطی
* ذخیره موقت داده خام

## نباید

* مدل‌های Django را Import کند
* `Product` ایجاد یا ویرایش کند
* `ProductVariant` ایجاد یا ویرایش کند
* `SupplierOffer` ایجاد یا ویرایش کند
* دیتابیس فروشگاه را تغییر دهد
* قوانین قیمت‌گذاری فروشگاه را اجرا کند
* درباره ایجاد یا بروزرسانی محصول تصمیم بگیرد

# 2. Extractors

## مسئولیت

تبدیل داده خام تأمین‌کننده به داده استاندارد قابل استفاده توسط سیستم بازبیا.

Extractor ساختار اختصاصی تأمین‌کننده را می‌شناسد، اما نباید دیتابیس فروشگاه را بشناسد.

## ورودی

داده خام دریافت‌شده توسط Fetcher، مانند:

* HTML
* JSON
* داده API
* فایل CSV یا Excel
* اطلاعات استخراج‌شده از صفحه محصول

## خروجی

یک شیء استاندارد مانند `ProductData`.

نمونه مفهومی:

```python
ProductData(
    name="...",
    slug="...",
    description="...",
    category="...",
    price=0,
    quantity=0,
    images=[],
    videos=[],
    specifications=[],
    attributes=[],
    tags=[],
    supplier="abdi",
    supplier_url="...",
)
```

## نکته مهم

فیلد:

```text
ProductData.quantity
```

موجودی گزارش‌شده توسط تأمین‌کننده است.

این فیلد به `Product.quantity` ارتباطی ندارد و نباید حذف شود.

## نباید

* مدل‌های Django را Import کند
* داده‌ای در دیتابیس ذخیره کند
* `Product` یا `SupplierOffer` ایجاد کند
* قوانین قیمت فروش را اجرا کند
* تأمین‌کننده مرجع واریانت را انتخاب کند

# 7. نمایش محصول

کنترل نمایش محصول در فروشگاه فقط بر عهده این فیلد است:

```python
Product.is_active
```

Fetcher و Sync تأمین‌کنندگان نباید مقدار `Product.is_active` را بدون یک قانون تجاری صریح تغییر دهند.

وجود موجودی نزد تأمین‌کننده به‌تنهایی به معنای نمایش محصول در فروشگاه نیست.

همچنین صفرشدن موجودی یک Offer نباید به‌صورت خودکار باعث غیرفعال‌شدن `Product` شود.

# 8. موجودی Product

فیلد زیر در مدل `Product` نباید وجود داشته باشد:

```python
Product.quantity
```

موجودی در سطح واریانت مدیریت می‌شود:

```python
ProductVariant.stock
```

موجودی کلی محصول باید از مجموع موجودی واریانت‌های آن محاسبه شود.

نمونه Query:

```python
.annotate(
    total_stock=Coalesce(
        Sum("variants__stock"),
        Value(0),
    )
)
```

محاسبه موجودی محصول نباید با اجرای Query جداگانه داخل حلقه انجام شود.

## تفاوت با ProductData.quantity

فیلد:

```python
ProductData.quantity
```

باید باقی بماند، زیرا موجودی گزارش‌شده توسط تأمین‌کننده را نگه می‌دارد.

جریان فعلی:

```text
ProductData.quantity
        ↓
SupplierOffer.supplier_stock
        ↓
Service هماهنگ‌سازی موجودی
        ↓
ProductVariant.stock
```

# 9. رابطه ProductVariant و SupplierOffer

هر `SupplierOffer` فقط به یک `ProductVariant` متصل است.

هر `ProductVariant` می‌تواند چند Offer از تأمین‌کنندگان مختلف داشته باشد:

```text
Product
└── ProductVariant
    ├── SupplierOffer A
    ├── SupplierOffer B
    └── SupplierOffer C
```

برای هر ترکیب زیر فقط یک Offer مجاز است:

```text
supplier + variant
```

این قانون بهتر است با Constraint دیتابیس نیز تضمین شود.

Offerهای تأمین‌کنندگان مختلف لازم نیست نام، لینک یا کد محصول یکسان داشته باشند.

نقطه اشتراک آن‌ها اتصال به یک `ProductVariant` داخلی بازبیا است.

---

# 10. شناسه محصول نزد تأمین‌کننده

فیلدهای زیر مربوط به هویت کالا در سیستم همان تأمین‌کننده هستند:

```text
supplier_product_name
supplier_product_code
supplier_url
```

قاعده پیشنهادی شناسایی Offer قبلی:

1. ابتدا ترکیب `supplier + supplier_product_code`
2. در صورت نبود کد معتبر، `supplier + supplier_url`
3. نام محصول فقط برای نمایش، جست‌وجوی کمکی یا بررسی انسانی استفاده شود

نام محصول نباید شناسه قطعی Offer باشد.

کد یکسان نزد دو تأمین‌کننده مختلف الزاماً به یک کالا اشاره نمی‌کند.

## وضعیت فعلی

تا زمانی که همه تأمین‌کنندگان `supplier_product_code` معتبر ارائه نمی‌کنند، شناسایی با این ترکیب قابل انجام است:

```text
supplier + supplier_url
```

معماری باید امکان مهاجرت به شناسایی بر اساس کد تأمین‌کننده را حفظ کند.

# 11. موجودی تأمین‌کننده و اجازه تأمین

این دو مفهوم مستقل هستند:

```text
supplier_stock
```

موجودی واقعی گزارش‌شده توسط تأمین‌کننده است.

```text
is_available
```

تصمیم تجاری بازبیا درباره مجازبودن استفاده از آن Offer است.

بنابراین ممکن است:

```text
supplier_stock > 0
is_available = False
```

دلایل احتمالی:

* قیمت نامناسب
* مشکلات همکاری
* کیفیت پایین خدمات
* تأخیر در ارسال
* غیرفعال‌سازی دستی
* محدودیت موقت فروش
* قوانین تجاری آینده

`is_available` نباید به‌صورت اجباری و در همه شرایط فقط از روی `supplier_stock` محاسبه شود.

ممکن است هنگام ایجاد اولیه Offer، مقدار پیش‌فرض آن بر اساس موجودی تعیین شود؛ اما پس از آن باید به‌عنوان یک تصمیم مستقل تجاری قابل تغییر باشد.

# 12. تأمین‌کننده اصلی

فیلد:

```text
is_primary
```

فقط نشان‌دهنده تأمین‌کننده پیش‌فرض یا ترجیحی است.

`is_primary` به‌تنهایی نباید داخل Serviceهای پایین‌سطح قیمت و موجودی بررسی شود.

برای مثال این منطق نباید مستقیماً داخل `variant_price_sync.py` قرار بگیرد:

```python
if not offer.is_primary:
    return False
```

زیرا ممکن است یک واریانت هیچ Offer اصلی نداشته باشد و در نتیجه همگام‌سازی آن به‌طور کامل متوقف شود.

## مسئول انتخاب Offer

انتخاب Offer مجاز یا مرجع باید در یک Policy یا Service مستقل انجام شود.

نمونه نام‌های احتمالی:

```python
can_sync_variant_from_offer(offer)
select_primary_offer(variant)
get_price_source_offer(variant)
get_stock_source_offers(variant)
```

قوانین آینده می‌توانند Offer را بر اساس موارد زیر انتخاب کنند:

* تأمین‌کننده اصلی
* ارزان‌ترین قیمت
* مطمئن‌ترین تأمین‌کننده
* بیشترین موجودی
* امتیاز ترکیبی
* انتخاب دستی
* فعال‌بودن Supplier
* زمان آخرین بررسی

# 13. سیاست فعلی Offer Sync

یک لایه مستقل برای سیاست هماهنگ‌سازی Offer در نظر گرفته شده است.

نمونه فایل:

```text
suppliers/services/offer_sync_policy.py
```

نمونه تابع:

```python
def can_sync_variant_from_offer(offer) -> bool:
    return True
```

## رفتار فعلی

در نسخه فعلی، تمام Offerها اجازه هماهنگ‌سازی قیمت و موجودی واریانت را دارند.

یعنی Policy فعلاً همیشه:

```python
True
```

برمی‌گرداند.

هدف از ایجاد این Service آن است که در آینده قوانین انتخاب Offer فقط در یک محل تغییر کنند و Serviceهای قیمت و موجودی نیازی به بازنویسی نداشته باشند.

## نکته

سیاست فعلی موقت است و برای پشتیبانی کامل از چند تأمین‌کننده کافی نیست.

طراحی کامل Policy بعداً انجام خواهد شد.

# 14. جداسازی قیمت و موجودی

```text
suppliers/services/
├── variant_price_sync.py
├── variant_stock_sync.py

```

```text
variant_price_sync.py
```

مسئولیت‌ها:

## Service موجودی

مسئولیت‌ها:

این Service نباید قیمت را تغییر دهد.

## مزیت جداسازی

ممکن است در آینده:

* قیمت از یک تأمین‌کننده گرفته شود
* موجودی از چند تأمین‌کننده جمع شود
* قیمت فقط از Offer اصلی گرفته شود
* موجودی بر اساس Offerهای قابل استفاده محاسبه شود

بنابراین قیمت و موجودی نباید به یک تصمیم واحد وابسته باشند.

# 16. جریان بروزرسانی Offer

جریان فعلی بروزرسانی باید به این شکل باشد:

```text
ProductData جدید
        ↓
پیدا کردن SupplierOffer
        ↓
مقایسه قیمت و موجودی
        ↓
در صورت تغییر قیمت:
    ثبت SupplierPriceHistory
        ↓
بروزرسانی SupplierOffer
        ↓
در صورت تغییر قیمت:
    sync_variant_price_from_offer()
        ↓
در صورت تغییر موجودی:
    sync_variant_stock_from_offer()
```

نمونه مسئولیت تابع `update_offer()`:

* تبدیل داده‌های ورودی به نوع مناسب
* تشخیص تغییر قیمت
* تشخیص تغییر موجودی
* تعیین `update_fields`
* ثبت تاریخچه در صورت تغییر قیمت
* ذخیره Offer
* فراخوانی Service قیمت در صورت تغییر قیمت
* فراخوانی Service موجودی در صورت تغییر موجودی
* بروزرسانی زمان‌های بررسی

`update_offer()` هماهنگ‌کننده است و نباید منطق داخلی محاسبه قیمت یا موجودی را پیاده‌سازی کند.

# 17. موجودی قابل فروش

موجودی قابل فروش یک واریانت باید توسط Service محاسبه شود.

نمونه نام Service:

```python
get_available_stock(variant)
```

منابع موجودی ممکن است شامل موارد زیر باشند:

```text
موجودی داخلی بازبیا
+
موجودی Offerهای قابل استفاده
```

Offer قابل استفاده حداقل باید شرایط زیر را داشته باشد:

```text
is_available = True
```

# 20. زمان‌های بررسی

```text
last_checked
```

آخرین زمانی است که سیستم Offer را بررسی کرده است؛ حتی اگر محصول پیدا نشده باشد.

```text
last_seen
```

آخرین زمانی است که محصول واقعاً در منبع تأمین‌کننده دیده شده است.

قانون بروزرسانی:

* در هر بررسی، `last_checked` بروزرسانی شود
* فقط در صورت پیدا شدن واقعی محصول، `last_seen` بروزرسانی شود
* پیدا نشدن محصول نباید فوراً باعث حذف Offer شود
* پیدا نشدن محصول نباید فوراً باعث غیرفعال‌شدن Product شود
* غیرفعال‌سازی یا حذف Offer باید بر اساس قانون جداگانه انجام شود

نمونه:

```python
checked_at = timezone.now()

offer.last_checked = checked_at

if product_was_found:
    offer.last_seen = checked_at
```

# 21. محصول جدید تأمین‌کننده

سیاست فعلی سیستم:

```text
محصول جدید تأمین‌کننده
        ↓
ایجاد Product
        ↓
ایجاد ProductVariant
        ↓
ایجاد SupplierOffer
        ↓
ثبت اولین تاریخچه قیمت
```

در جریان فعلی، اطلاعات اولیه قیمت و موجودی هنگام ایجاد واریانت از `ProductData` دریافت می‌شوند.

## نکته معماری

اجرای Policy پس از ایجاد واریانت، طراحی نهایی و کامل محسوب نمی‌شود.

برای طراحی کامل باید دو نوع Policy جدا در نظر گرفته شود:

```text
Product Creation Policy
Offer Sync Policy
```

### Product Creation Policy

قبل از ایجاد Product و ProductVariant بررسی می‌کند که آیا داده تأمین‌کننده اجازه ساخت محصول جدید را دارد یا نه.

### Offer Sync Policy

مشخص می‌کند یک Offer موجود اجازه دارد قیمت یا موجودی واریانت را تغییر دهد یا نه.

## وضعیت فعلی

طراحی و پیاده‌سازی کامل این دو Policy فعلاً کنار گذاشته شده است تا بعداً همراه با پشتیبانی چند تأمین‌کننده بازطراحی شود.

بنابراین جریان فعلی ایجاد محصول بدون تغییر اساسی ادامه پیدا می‌کند.

```python
def can_sync_variant_from_offer(offer):
```
