هر لایه فقط یک مسئولیت مشخص دارد و نباید وارد مسئولیت لایه‌های دیگر شود.

---

1. fetchers

مسئولیت

برقراری ارتباط با سایت تأمین‌کننده و استخراج اطلاعات.

این بخش تنها جایی است که با اینترنت، HTML و ساختار سایت تأمین‌کننده سروکار دارد.

---

ورودی

- آدرس محصول
- آدرس دسته‌بندی
- API تأمین‌کننده (در صورت وجود)

---

خروجی

داده خام یا JSON استاندارد مانند:

{
    "name": "...",
    "price": "...",
    "stock": "...",
    "url": "...",
    "images": [],
    "description": "...",
    "specifications": []
}

---

مجاز است

- ارسال Request
- دریافت Response
- استفاده از requests
- استفاده از BeautifulSoup
- استخراج HTML
- تبدیل HTML به JSON

---

نباید

- مدل‌های Django را Import کند.
- Product ایجاد کند.
- SupplierOffer ایجاد کند.
- دیتابیس را تغییر دهد.
- قوانین قیمت‌گذاری را بداند.
- منطق فروشگاه را بداند.

---

2. sync

مسئولیت

تبدیل JSON استاندارد به اطلاعات قابل ذخیره در دیتابیس.

این بخش مغز عملیات همگام‌سازی است.

---

ورودی

JSON استاندارد تولیدشده توسط Fetcher

---

خروجی

- ایجاد محصول
- بروزرسانی محصول
- بروزرسانی موجودی
- بروزرسانی قیمت
- ثبت تاریخچه قیمت

---

وظایف

- پیدا کردن SupplierOffer
- بررسی وجود محصول
- مقایسه اطلاعات قبلی و جدید
- ثبت تغییرات
- ثبت تاریخچه قیمت
- ثبت Log
- تصمیم‌گیری برای Create یا Update

---

مجاز است

- استفاده از ORM جنگو
- خواندن و نوشتن دیتابیس
- استفاده از Transaction
- ثبت Log

---

نباید

- HTML بخواند.
- Request اینترنتی ارسال کند.
- BeautifulSoup استفاده کند.
- ساختار سایت تأمین‌کننده را بشناسد.

---

3. services

مسئولیت

پیاده‌سازی قوانین کسب‌وکار (Business Logic)

تمام قوانینی که مستقل از تأمین‌کننده هستند در این قسمت قرار می‌گیرند.

---

نمونه وظایف

- ایجاد Supplier
- ایجاد Product
- ایجاد SupplierOffer
- ایجاد Variant
- قوانین قیمت‌گذاری
- محاسبه سود
- قوانین فعال یا غیرفعال بودن پیشنهاد
- اعتبارسنجی اطلاعات

---

ویژگی مهم

Service نباید بداند اطلاعات از کدام سایت آمده است.

برای Service هیچ تفاوتی ندارد داده‌ها از:

- پخش عبدی
- تأمین‌کننده دوم
- فایل CSV
- API
- فایل Excel

آمده باشند.

فقط داده استاندارد دریافت می‌کند و منطق کسب‌وکار را اجرا می‌کند.

---

وابستگی بین لایه‌ها

fetchers

وابستگی برعکس ممنوع است.

یعنی:

- Service نباید Fetcher را Import کند.
- Fetcher نباید Model را Import کند.
- Model نباید Fetcher را بشناسد.

---

قانون طلایی پروژه

هر فایل فقط یک مسئولیت دارد.

هر پوشه فقط یک هدف دارد.

هیچ بخشی نباید مسئولیت بخش دیگر را انجام دهد.

اگر هنگام توسعه احساس شد یک فایل هم Request می‌زند، هم HTML را پردازش می‌کند، هم قیمت را محاسبه می‌کند و هم در دیتابیس ذخیره می‌کند، یعنی معماری نقض شده و آن فایل باید به چند بخش کوچک‌تر تقسیم شود.
