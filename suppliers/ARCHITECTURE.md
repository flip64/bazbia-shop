# معماری ماژول Suppliers

## هدف

ماژول `suppliers` مسئول ارتباط با تأمین‌کنندگان، دریافت اطلاعات محصولات، استانداردسازی داده‌ها و همگام‌سازی آن‌ها با دیتابیس فروشگاه بازبیا است.

این سند تصمیم‌های قطعی معماری این ماژول را ثبت می‌کند تا توسعه‌های بعدی با منطق یکسان انجام شوند.

---

## ساختار کلی جریان داده

```text
Website / API / File
        ↓
Fetcher
        ↓
Extractor
        ↓
Standard Data
        ↓
Sync
        ↓
Services
        ↓
Django Models
```

هر لایه فقط یک مسئولیت مشخص دارد و نباید وارد مسئولیت لایه‌های دیگر شود.

---

## 1. Fetchers

### مسئولیت

برقراری ارتباط با سایت یا منبع داده تأمین‌کننده و استخراج اطلاعات.

این بخش تنها جایی است که با اینترنت، HTML، API، فایل و ساختار اختصاصی تأمین‌کننده سروکار دارد.

### ورودی‌های احتمالی

- آدرس محصول
- آدرس دسته‌بندی
- API تأمین‌کننده
- فایل CSV، Excel یا JSON

### خروجی

داده استاندارد، مانند:

```json
{
  "name": "...",
  "supplier_product_code": "...",
  "price": 0,
  "stock": 0,
  "url": "...",
  "images": [],
  "description": "...",
  "specifications": []
}
```

### مجاز است

- ارسال Request
- دریافت Response
- استفاده از `requests`
- استفاده از `BeautifulSoup`
- خواندن HTML یا API
- تبدیل داده خام به داده استاندارد

### نباید

- مدل‌های Django را Import کند
- `Product` ایجاد یا ویرایش کند
- `SupplierOffer` ایجاد یا ویرایش کند
- دیتابیس را تغییر دهد
- قوانین قیمت‌گذاری یا فروشگاه را اجرا کند

---

## 2. Sync

### مسئولیت

هماهنگ‌کردن داده استاندارد تأمین‌کننده با اطلاعات موجود در دیتابیس.

Sync مغز عملیات همگام‌سازی است، اما اجرای قوانین کسب‌وکار را به Serviceها واگذار می‌کند.

### ورودی

داده استاندارد تولیدشده توسط Fetcher.

### وظایف

- شناسایی محصول یا واریانت مرتبط
- پیدا کردن `SupplierOffer` قبلی
- تصمیم‌گیری میان Create و Update
- مقایسه قیمت و موجودی قبلی و جدید
- فراخوانی Service مناسب
- ثبت Log نتیجه عملیات
- مدیریت Transaction در عملیات چندمرحله‌ای

### نباید

- HTML بخواند
- Request اینترنتی ارسال کند
- از `BeautifulSoup` استفاده کند
- ساختار سایت تأمین‌کننده را بشناسد
- منطق تجاری را داخل خود تکرار کند

---

## 3. Services

### مسئولیت

پیاده‌سازی قوانین کسب‌وکار مستقل از منبع داده.

Service نباید بداند اطلاعات از پخش عبدی، تأمین‌کننده دیگر، CSV، Excel یا API آمده است.

### نمونه وظایف

- ایجاد یا بروزرسانی Supplier
- ایجاد Product و ProductVariant
- ایجاد یا بروزرسانی SupplierOffer
- ثبت تاریخچه قیمت
- محاسبه موجودی قابل فروش
- اعتبارسنجی داده
- اجرای قوانین فعال یا غیرفعال‌بودن پیشنهاد

---

## 4. Models

مدل‌ها ساختار و وضعیت فعلی داده را نگه می‌دارند. منطق پیچیده انتخاب تأمین‌کننده، محاسبه موجودی کل یا اجرای Workflow نباید در مدل‌ها پراکنده شود.

---

# قوانین قطعی معماری

## 5. نمایش محصول

کنترل نمایش محصول در فروشگاه فقط بر عهده این فیلد است:

```python
Product.is_active
```

Fetcher و Sync تأمین‌کنندگان نباید مقدار `Product.is_active` را بدون یک قانون تجاری صریح تغییر دهند.

وجود موجودی نزد تأمین‌کننده به‌تنهایی به معنای نمایش محصول در فروشگاه نیست.

---

## 6. رابطه ProductVariant و SupplierOffer

هر `SupplierOffer` فقط به یک `ProductVariant` متصل است.

هر `ProductVariant` می‌تواند چند Offer از تأمین‌کنندگان مختلف داشته باشد:

```text
Product
└── ProductVariant
    ├── SupplierOffer A
    ├── SupplierOffer B
    └── SupplierOffer C
```

برای هر ترکیب زیر فقط یک Offer مجاز است:

```text
supplier + variant
```

Offerهای تأمین‌کنندگان مختلف لازم نیست نام یا کد محصول یکسان داشته باشند. نقطه اشتراک آن‌ها اتصال به یک `ProductVariant` داخلی بازبیا است.

---

## 7. شناسه محصول نزد تأمین‌کننده

فیلدهای زیر مربوط به هویت کالا در سیستم همان تأمین‌کننده هستند:

```text
supplier_product_name
supplier_product_code
supplier_url
```

قاعده شناسایی Offer قبلی:

1. ابتدا ترکیب `supplier + supplier_product_code`
2. در صورت نبود کد معتبر، `supplier + supplier_url`
3. نام محصول فقط برای نمایش، جست‌وجوی کمکی یا بررسی انسانی استفاده شود و شناسه قطعی نباشد

کد یکسان نزد دو تأمین‌کننده مختلف الزاماً به یک کالا اشاره نمی‌کند.

---

## 8. موجودی تأمین‌کننده و اجازه تأمین

این دو مفهوم مستقل هستند:

```text
supplier_stock
```

موجودی واقعی گزارش‌شده توسط تأمین‌کننده است.

```text
is_available
```

تصمیم تجاری بازبیا درباره مجازبودن استفاده از آن Offer است.

بنابراین ممکن است:

```text
supplier_stock > 0
is_available = False
```

دلایل احتمالی:

- قیمت نامناسب
- مشکلات همکاری
- کیفیت پایین خدمات
- تأخیر در ارسال
- غیرفعال‌سازی دستی
- قوانین تجاری آینده

`is_available` نباید به‌صورت اجباری فقط از روی `supplier_stock` محاسبه شود.

---

## 9. تأمین‌کننده اصلی

```text
is_primary
```

فقط نشان‌دهنده تأمین‌کننده پیش‌فرض یا ترجیحی است و انتخاب اجباری محسوب نمی‌شود.

قوانین آینده می‌توانند Offer را بر اساس موارد زیر انتخاب کنند:

- تأمین‌کننده اصلی
- ارزان‌ترین قیمت
- مطمئن‌ترین تأمین‌کننده
- بیشترین موجودی
- امتیاز ترکیبی
- انتخاب دستی

سرویس انتخاب تأمین‌کننده در مرحله‌ای جداگانه پیاده‌سازی خواهد شد.

---

## 10. موجودی قابل فروش

موجودی قابل فروش یک واریانت از دو منبع تشکیل می‌شود:

```text
موجودی انبار خود بازبیا
+
مجموع موجودی Offerهای قابل استفاده
```

Offer قابل استفاده باید حداقل این شرایط را داشته باشد:

```text
is_available = True
supplier_stock > 0
```

موجودی کل نباید به‌عنوان یک فیلد جداگانه در دیتابیس ذخیره شود؛ باید هنگام نیاز توسط Service محاسبه شود.

نمونه نام سرویس:

```python
get_available_stock(variant)
```

یک واریانت زمانی قابل فروش است که حداقل یکی از شرایط زیر برقرار باشد:

- موجودی انبار خود بازبیا بیشتر از صفر باشد
- حداقل یک `SupplierOffer` قابل استفاده و دارای موجودی داشته باشد

وجود SupplierOffer برای واریانتی که در انبار بازبیا موجود است اجباری نیست.

---

## 11. تاریخچه قیمت

`SupplierOffer.purchase_price` قیمت جاری تأمین‌کننده را نگه می‌دارد.

`SupplierPriceHistory` تاریخچه قیمت را به‌صورت Snapshot نگه می‌دارد:

```text
supplier_offer
price
created_at
```

قانون ثبت:

- اگر قیمت تغییر نکرد، رکورد تاریخچه جدید ساخته نشود
- اگر قیمت تغییر کرد، قیمت جدید در تاریخچه ثبت شود
- سپس `purchase_price` با مقدار جدید بروزرسانی شود

قیمت قبلی از Snapshot قبلی قابل استخراج است؛ بنابراین ذخیره هم‌زمان `old_price` و `new_price` لازم نیست.

---

## 12. تاریخچه موجودی

فعلاً تاریخچه موجودی تأمین‌کننده نگهداری نمی‌شود.

فقط مقدار فعلی زیر بروزرسانی می‌شود:

```text
SupplierOffer.supplier_stock
```

در صورت ایجاد نیاز واقعی، مدل `SupplierStockHistory` بعداً قابل اضافه‌شدن است.

---

## 13. زمان‌های بررسی

```text
last_checked
```

آخرین زمانی است که سیستم Offer را بررسی کرده است؛ حتی اگر محصول پیدا نشده باشد.

```text
last_seen
```

آخرین زمانی است که محصول واقعاً در منبع تأمین‌کننده دیده شده است.

قانون بروزرسانی:

- در هر بررسی، `last_checked` بروزرسانی شود
- فقط در صورت پیدا شدن واقعی محصول، `last_seen` بروزرسانی شود
- پیدا نشدن محصول نباید فوراً باعث حذف Offer شود
- غیرفعال‌سازی یا حذف Offer باید بر اساس قانون جداگانه انجام شود

---

## 14. محصول جدید تأمین‌کننده

سیاست فعلی سیستم:

```text
محصول جدید تأمین‌کننده
→ ایجاد Product
→ ایجاد ProductVariant
→ ایجاد SupplierOffer
```

در آینده می‌توان سیاست برخورد با محصول جدید را برای هر Supplier قابل تنظیم کرد، مانند:

```text
auto_create
report_only
ignore
```

این قابلیت فعلاً پیاده‌سازی نمی‌شود، اما معماری باید امکان اضافه‌کردن آن را حفظ کند.

# وابستگی میان لایه‌ها

```text
fetchers
   ↓
sync
   ↓
services
   ↓
models
```

وابستگی برعکس ممنوع است:

- Service نباید Fetcher را Import کند
- Fetcher نباید Model را Import کند
- Model نباید Fetcher یا Sync را بشناسد
- منطق مشترک نباید در اسکرپر اختصاصی یک تأمین‌کننده هاردکد شود

# قابلیت‌های موکول‌شده به آینده

موارد زیر تصمیم‌گیری شده‌اند اما فعلاً پیاده‌سازی نمی‌شوند:

- سیاست اختصاصی هر Supplier برای محصول جدید
- سرویس انتخاب تأمین‌کننده
- انتخاب ارزان‌ترین یا مطمئن‌ترین Offer
- امتیازدهی تأمین‌کنندگان
- تاریخچه موجودی
- محدودکردن بخشی از موجودی تأمین‌کننده برای فروش

# قانون طلایی پروژه

هر فایل فقط یک مسئولیت دارد و هر پوشه فقط یک هدف مشخص دارد.

اگر یک فایل هم‌زمان Request ارسال کند، HTML پردازش کند، قوانین تجاری را اجرا کند و دیتابیس را تغییر دهد، معماری نقض شده است و آن فایل باید به چند بخش مستقل تقسیم شود.
